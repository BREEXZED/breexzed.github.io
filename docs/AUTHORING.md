# Authoring The Estate

This is the practical guide for living inside the estate: how to publish, how to update, and where to place text, images, links, and code.

## 1. Where content lives

- All public estate content lives in `nodes/`.
- Each markdown file in `nodes/` becomes one node in the estate.
- The route for that node becomes `/node/:id`.
- The main public surfaces (`/`, `/corpus`, `/map`, `/signals`, `/projects`) are discovery layers.
- The dedicated reading surface is always the node page at `/node/:id`.

## 2. Which template to start from

Copy one file from `templates/` and rename it into the right folder under `nodes/`.

- `templates/concept.md`
- `templates/articulation.md`
- `templates/signal.md`
- `templates/trail.md`
- `templates/projects.md`

The seeded content is only there as current demo content. You can delete or replace it node by node when you are ready.

## 3. The minimum publishing workflow

1. Copy the right template into `nodes/<type>/your-file.md`.
2. Fill in the frontmatter.
3. Write the body in normal markdown.
4. Run `npm run build:content`.
5. Run `npm run smoke` when you want the quick browser verification too.

If the build passes, the estate has accepted the node.

## 4. What frontmatter does

Core fields:

```yaml
id: your_stable_node_id
title: Human Title
formula: relation -> meaning
depth: 1
type: concept
status: published
parent: root
children: []
connects: []
```

Important rules:

- `id` must be lowercase and stable.
- `children` is for structural movement through the estate.
- `connects` is for cross-links and graph relations.
- `source` is authored provenance.
- `sourcePath` is generated by the compiler. Do not author it manually.

## 5. How internal links work

Link to another node by linking to its markdown file.

Example:

```md
[Knowledge In Motion](../articulations/knowledge-in-motion.md)
```

The runtime resolves that to the canonical node page automatically.

That means:

- internal markdown links stay inside the estate
- graph clicks stay inside the estate
- corpus cards stay inside the estate
- search results stay inside the estate

## 6. How to add images

You have two good options.

### Inline image in the body

Put the image near the node file and reference it relatively in markdown.

```md
![Field sketch](./image-name.jpg)
```

During `npm run build:content`, the asset pipeline copies supported images into public assets and rewrites the path.

### Hero/supporting image from frontmatter

Use:

```yaml
visual: ./image-name.jpg
```

or

```yaml
thumbnail: ./image-name.jpg
```

The dedicated node page will render that image as a large visual block.

Supported asset extensions:

- `.png`
- `.jpg`
- `.jpeg`
- `.gif`
- `.svg`
- `.webp`
- `.avif`

## 7. How to add code snippets

Use fenced markdown code blocks.

```md
```ts
export function helloEstate() {
  return 'hello';
}
```
```

The node page renders code blocks in a dedicated code shell, and project nodes use the same markdown rendering.

Use language tags when you can:

- `ts`
- `js`
- `tsx`
- `bash`
- `json`
- `md`

## 8. How to add external links

Use normal markdown links.

```md
[Reference](https://example.com)
```

Use external links only when the thing genuinely lives outside the estate.

If the resource should remain part of the estate, make it another node and link the markdown file instead.

## 9. How each node type should feel

`concept`

- durable lens
- framing idea
- organizing thought

`articulation`

- shaped expression
- public statement
- synthesis or claim

`signal`

- something encountered in the world
- still live, still moving
- something being tracked

`trail`

- the trace of contact
- note, margin, fragment, confirming evidence

`projects`

- builds, experiments, products, implementations
- can include images, code, links, and external references

## 10. What catches edge cases now

The compiler already catches:

- duplicate IDs
- missing required fields
- invalid node types
- invalid signal status values
- broken `parent` references
- broken `children` references
- broken `connects` references
- broken internal markdown links

The smoke layer now checks:

- `/map` graph default
- `/node/:id` dedicated node page
- search-to-node routing
- internal markdown link routing

## 11. Social and account config

Edit:

- `src/config/site.ts`

That file controls:

- GitHub
- Mail
- X
- Spotify
- YouTube when you are ready

If a value is empty or `null`, it will not render in the footer.

## 12. The mental model

Think of the estate like this:

- `nodes/` is where you write reality into the system
- the build pipeline turns it into a coherent public topology
- discovery surfaces help people arrive
- node pages are where they actually stay and read
- the map is for relation
- the node page is for encounter
